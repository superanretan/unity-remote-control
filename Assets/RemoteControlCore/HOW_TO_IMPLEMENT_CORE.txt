# SuperAnretan.RemoteControl — Implementation Guide (Core System)

This document explains how to implement the Remote Control system (Controller <-> Host)
in any new project using RemoteControlCore. The architecture is based on EventChannels
(ScriptableObjects) and `RemoteCommand` data objects sent over Unity Transport.

---

## 0. Installation in a New Project (Unity Package Manager — SSH)

To use this module in another application, make sure your machine has SSH access
configured for your GitHub account.

Open **Window → Package Manager** in your target project, click **+** and choose
**"Add package from git URL…"**. Paste the SSH link:

    git@github.com:SuperAnretan/unity-remote-control.git?path=/Assets/RemoteControlCore#v1.1.0

*(Replace `v1.1.0` with the current release tag and adjust the URL for your repo.)*

### Required dependency (must already be in the target project)

| Package                  | Minimum version | Why                         |
|--------------------------|-----------------|-----------------------------|
| `com.unity.transport`    | 2.4.0           | TCP network transport       |
| `com.unity.ugui`         | 2.0.0           | Canvas UI (buttons, etc.)   |
| TextMeshPro              | (built-in)      | TMP_Dropdown, TMP text      |

## 1. Initial Scene Setup (Plug & Play)

### Controller scene (e.g. mobile / desktop app)

1. Create an empty GameObject → add `TransportClient` component.
   - Assign the SO channels from `DefaultSetup/SO/` (see section 4 for the full list).
2. Add the **`NetworkDiscoveryPanel`** prefab to your Canvas
   (`Runtime/UI/Prefabs/NetworkDiscoveryPanel`).
   - This single prefab contains everything the client needs for networking UI:
     **host discovery dropdown**, **Connect / Disconnect buttons**, **status indicator**,
     and a **scrolling log area**.
   - It already has `NetworkDiscoveryClient` on itself — no extra components needed.
   - All SO event channels from `DefaultSetup/SO/` are pre-wired in the prefab.

### Host scene (e.g. VR headset)

1. Create an empty GameObject → add:
   - `TransportHost`
   - `NetworkDiscoveryHost`
   - `CommandProcessor`
2. Assign the SO channels and registries from `DefaultSetup/SO/`.

Both sides share the same `NetworkConfig` asset (port 7777 for TCP, port 7778 for UDP
discovery). No code changes needed — just assign the SOs.

## 2. How to SEND a Command (Controller Side)

To send any action from UI to the Host, create a `RemoteCommand` and raise it on the
command send channel.

**Step 1:** Add a reference to `CommandEventChannel` in your UI script.
**Step 2:** Create a `RemoteCommand` with these parameters:
  1. `commandType` (string) — unique action name, e.g. `"set_color"`, `"spawn_enemy"`.
  2. `targetId` (string) — unique ID of the target object on the Host scene, e.g. `"player_1"`.
  3. `value` (string) — primary data, e.g. `"#FF0000"`, a number, or plain text.
  4. `payload` (string, optional) — extra JSON blob for complex payloads.
**Step 3:** Call `.Raise(cmd)` on the channel.

**Example:**
```csharp
using UnityEngine;
using SuperAnretan.RemoteControl;

public class MyCustomUI : MonoBehaviour
{
    // Assign CommandSendChannel from DefaultSetup/SO in the Inspector
    [SerializeField] private CommandEventChannel commandSendChannel;

    public void SendJumpCommand()
    {
        // Command "jump" targeting object with ID "player_1", force value "10.5"
        var cmd = new RemoteCommand("jump", "player_1", "10.5");
        commandSendChannel.Raise(cmd);
    }
}
```

## 3. How to RECEIVE a Command (Host Side)

The receiving side has two elements: a **Target** and a **Handler**.

### A. Configuring the Target (CommandTarget)

The object you want to control (e.g. the player character) needs a unique `TargetId`.

1. Add `CommandTarget` component to the scene object.
2. Set its `Target Id` (e.g. `"player_1"`).
3. Assign the `Target Registry` SO from `DefaultSetup/SO/`.

This lets the system resolve which GameObject an incoming command refers to.

### B. Configuring the Logic (CommandHandler)

Create a script that listens for a specific command name (e.g. `"jump"`) and decides
what to do when it arrives.

1. Create a new script inheriting from `CommandHandlerBase`.
2. Override `CommandType` — return the command name string.
3. Override `Handle(RemoteCommand command, GameObject target)`.
4. Place this script anywhere in the scene and assign the `Handler Registry` SO.

**Example Handler:**
```csharp
using UnityEngine;
using SuperAnretan.RemoteControl;

public class JumpCommandHandler : CommandHandlerBase
{
    // Which command do we react to?
    public override string CommandType => "jump";

    // What do we do when the command arrives?
    public override void Handle(RemoteCommand command, GameObject target)
    {
        if (float.TryParse(command.value, out float jumpForce))
        {
            Rigidbody rb = target.GetComponent<Rigidbody>();
            if (rb != null)
            {
                rb.AddForce(Vector3.up * jumpForce, ForceMode.Impulse);
                Debug.Log($"Object {target.name} jumped with force {jumpForce}");
            }
        }
    }
}
```

## 4. Event Channel Architecture

Instead of hard references (spaghetti code), this system uses ScriptableObjects for all
cross-component communication. All pre-made channels live in:

    Assets/RemoteControlCore/Runtime/DefaultSetup/SO/

### Channel reference table

| Asset name                      | Type                  | Purpose                                                   |
|---------------------------------|-----------------------|-----------------------------------------------------------|
| `CommandSendChannel`            | CommandEventChannel   | Controller raises this to send commands to the host.      |
| `CommandReceivedChannel`        | CommandEventChannel   | Host receives deserialized commands here (CommandProcessor). |
| `ConnectRequestChannel`         | StringEventChannel    | Raise with an IP string to make TransportClient connect.  |
| `DisconnectRequestChannel`      | VoidEventChannel      | Raise to force TransportClient to disconnect.             |
| `OnConnectedChannel`            | VoidEventChannel      | Fires when TransportClient successfully connects.         |
| `OnDisconnectedChannel`         | VoidEventChannel      | Fires when TransportClient disconnects.                   |
| `OnHostSelectedChannel`         | StringEventChannel    | Returns the IP of the host selected in the discovery dropdown. |
| `OnClientConnectedChannel`      | VoidEventChannel      | Host-side: fires when a client connects.                  |
| `OnClientDisconnectedChannel`   | VoidEventChannel      | Host-side: fires when a client disconnects.               |
| `LogChannel`                    | StringEventChannel    | Send strings here to display in the on-screen log (DebugLogUI or NetworkDiscoveryPanel). |
| `NetworkConfig`                 | NetworkConfig         | Shared config: TCP port (7777), UDP discovery port (7778), max connections, fallback host IP. |
| `HandlerRegistry`               | CommandHandlerRegistry| Runtime registry of all active command handlers.          |
| `TargetRegistry`                | CommandTargetRegistry | Runtime registry of all active command targets.           |

## 5. NetworkDiscoveryPanel (Controller UI Prefab)

`Runtime/UI/Prefabs/NetworkDiscoveryPanel` is a self-contained prefab that provides
the complete client-side networking UI in a single drag-and-drop.

### What it includes

| Element           | Function                                                       |
|-------------------|----------------------------------------------------------------|
| Host Dropdown     | TMP_Dropdown populated by UDP discovery (auto-scans on enable) |
| Refresh button    | Restarts discovery scan                                        |
| Connect button    | Raises `ConnectRequestChannel` with the selected host IP       |
| Disconnect button | Raises `DisconnectRequestChannel`                              |
| Status indicator  | Shows connection state: 🔴 Disconnected / 🟠 Connecting / 🟢 Connected |
| Log area          | Timestamped scrolling log (listens to `LogChannel`)            |

### Components on the root GameObject

| Component                | Role                                      |
|--------------------------|-------------------------------------------|
| `NetworkDiscoveryClient` | Listens for UDP broadcast from host        |
| `NetworkDiscoveryPanel`  | Wires all UI elements to SO event channels |

### Automatic behavior

- Buttons auto-enable/disable based on connection state.
- Dropdown and Refresh lock while connected (prevents accidental changes).
- Discovery starts automatically when the panel becomes active.
- Falls back to `NetworkConfig.HostIP` if no host is discovered.

### Prefab hierarchy

```
NetworkDiscoveryPanel         (Image, VLG, ContentSizeFitter, NetworkDiscoveryClient, NetworkDiscoveryPanel)
├── Title                     (TextMeshProUGUI — "Network Discovery")
├── StatusText                (TextMeshProUGUI — richText status indicator)
├── Separator1                (Image, 1px)
├── HostDropdown              (TMP_Dropdown)
├── ButtonRow                 (HorizontalLayoutGroup)
│   ├── RefreshButton         (Button)
│   ├── ConnectButton         (Button)
│   └── DisconnectButton      (Button)
├── Separator2                (Image, 1px)
└── LogArea                   (Image, dark background)
    └── LogText               (TextMeshProUGUI — scrolling timestamped log)
```

## 6. Debug Log Panel (Optional)

`Runtime/Debug/DebugLogUI` is a simpler, standalone log display. It listens to any
`StringEventChannel` (typically `LogChannel`) and renders the last N messages in a
TextMeshProUGUI. Use this if you want a separate, always-visible debug console
independent of the NetworkDiscoveryPanel.

Note: `NetworkDiscoveryPanel` already has its own built-in log area, so `DebugLogUI`
is only needed if you want a second log view or if you're on the Host side.

## 7. Tips for AI / LLM Implementation

1. **Always use Event Channels** for network communication. Never use
   `FindObjectOfType<TransportClient>` or similar in your code.
2. **One Handler = one feature.** E.g. `SetColorHandler`, `JumpHandler`,
   `SpawnHandler`. Handlers don't need to be on the target object — they can be
   global (e.g. on a manager object), because `Handle()` always receives the
   resolved `GameObject target` as a parameter.
3. **Discovery runs on UDP port 7778**, TCP transport on **port 7777**. The Host
   automatically broadcasts its IP on the local network. The Controller picks it
   up through `NetworkDiscoveryClient` and displays it in the dropdown.
4. **RemoteCommand has 4 fields:** `commandType`, `targetId`, `value`, and
   `payload`. Use `value` for simple data (a color hex, a number) and `payload`
   for complex JSON blobs. Both are strings.
5. **DiscoveryUI.cs is deprecated.** Use `NetworkDiscoveryPanel` instead — it
   replaces `DiscoveryUI` and adds Connect/Disconnect/Status/Log functionality.
